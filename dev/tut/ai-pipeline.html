AI Assessment Pipeline

The technical differentiator of Cervantes: seven chained AI phases transform a raw rubric file into fully playable Visual Novel assessment arcs with real-time Socratic dialogue and automated signal extraction.

Overview

Two Models, One Pipeline

Two AI models work in tandem throughout the pipeline, each chosen for what it does best. Curriculum alignment is handled by a specialist model; everything creative and analytical goes through Gemini.

CurricuLLM-AU

api.curricullm.com/v1

Curriculum-aligned rubric parser. Understands Australian curriculum standards and extracts structured learning outcomes, key concepts, and common misconceptions.

CurricuLLM-AU CurricuLLM-AU-VIC CurricuLLM-AU-WA CurricuLLM-NZ

Gemini 2.5 Flash

google.genai SDK

Scene generation, real-time Socratic dialogue, signal extraction, and arc endings. Used via the async Google GenAI SDK (google.genai.aio).

Deep Dive

Phase-by-Phase Breakdown

Phase 1

Rubric Parsing (CurricuLLM)

The professor uploads a rubric file (PDF, DOCX, or TXT). Text extraction is handled by PyPDF2 for PDFs and python-docx for Word documents. The extracted text is sent to the CurricuLLM-AU model via an OpenAI-compatible API.

Input

  • Uploaded rubric file (PDF/DOCX/TXT)
  • Extracted raw text via PyPDF2 / python-docx

Output

  • learning_outcomes
  • key_concepts
  • misconceptions
  • common_student_misconceptions

Prompt file: curricullm/curricullm_rubric_parse.md

Fallback: If CurricuLLM is unreachable or returns an error, the system automatically retries with Gemini using the same prompt structure, ensuring the pipeline never stalls at step one.

Phase 2

Arc Planning (Gemini)

The parsed curriculum data from Phase 1 feeds into a dynamically assembled prompt in arc/service.py. Gemini produces a full arc structure: a sequence of 3-5 scenes, each with pedagogical targeting.

Input

  • Parsed curriculum JSON from Phase 1
  • Dynamic prompt built in arc/service.py

Output per Scene

  • Scene type (bridge / deep / side_event)
  • Concept target (learning objective)
  • Misconception target
  • Character archetype & personality traits
  • Socratic angles (questioning strategies)
Phase 3

Character Generation (Gemini)

Each scene's character archetype from the arc plan is expanded into a full character profile. A character pool of 2-3 name variants per scene provides anti-copying diversity — each student is randomly assigned a variant so dialogue screenshots cannot be reused.

Input

  • Scene character archetypes from Phase 2

Output per Character

  • name (2-3 variants)
  • role
  • personality_prompt
  • voice_register
  • sprite_set
Phase 4

Scene Content Generation (Gemini)

With the arc plan and characters in place, Gemini generates the full Visual Novel script for each scene. This is triggered when the professor clicks "Publish" on the class. The scene generation prompt is combined with annotation prompts for consistent formatting.

Prompt files:

  • system/scene_generation_v2.md — main generation prompt
  • annotations/emotion_tags.md — emotion formatting
  • annotations/action_tags.md — action formatting
  • annotations/formatting_tags.md — general formatting

Output format:

[narration] Context setting... [character:Name] [emotion] Dialogue line... [player_prompt] Question text... **A. Option text** -- branch **B. Option text** -- branch [freeform]
Phase 5

Real-time Socratic Dialogue (Gemini)

When a student submits an answer during scene play, the system sends a request to POST /api/dialogue/turn. Gemini receives the full conversation context and responds in character with Socratic pushback — it never gives answers directly, only asks probing follow-up questions.

Input per Turn

  • Student's answer text
  • Full conversation history
  • Scene context & concept target
  • Character personality prompt

Output

  • In-character Socratic response
  • Adapted character emotion
  • Turn count (max 5 per scene)

Prompt file: system/pushback_dialogue.md
Character emotion dynamically adapts based on the quality of student reasoning across turns.

Phase 6

Signal Extraction (Gemini)

Triggered automatically on scene completion. Gemini analyzes the full conversation history, the student's initial answer, and their revised answer to produce a structured assessment signal.

Input

  • Full conversation history
  • Initial student answer
  • Revised student answer

Output

  • Status: mastery | revised_with_scaffolding | critical_gap
  • Reasoning signals & reflection
  • Multipart response analysis

Prompt file: system/signal_extraction.md

Phase 7

Arc Ending Generation (Gemini)

After all scenes in an arc are completed, Gemini generates a personalized narrative ending based on the student's aggregated performance across every scene. The ending is written in the same VN style and stored in the arc_endings Firestore collection.

Input

  • Aggregated signals from all scenes
  • Overall performance level

Output

  • Personalized narrative ending
  • Performance reflection
  • Stored in arc_endings collection

Prompt file: system/arc_ending.md

Resilience

Error Handling & Recovery

Every AI call in the pipeline is designed to fail gracefully. The system uses multiple layers of protection to ensure a student never sees a broken experience.

R Retry with Backoff

Tenacity-based exponential backoff with up to 3 retry attempts on every API call. Handles transient network failures and rate limits.

F Model Fallback

If CurricuLLM fails during rubric parsing (Phase 1), the system automatically falls through to Gemini with the same prompt structure.

J JSON Recovery

Automatic repair of malformed JSON from model outputs — fixes unescaped newlines, trailing commas, and truncated responses.

A Fully Async

All AI operations use async clients (google.genai.aio, AsyncOpenAI) to prevent blocking the event loop during generation.
